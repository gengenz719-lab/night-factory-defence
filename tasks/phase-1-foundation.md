# 業務指示書: Phase 1 — 製品基盤の作り直し(WP1-1〜WP1-4)

発行: PM / 2026-07-11
対象リポジトリ: night-factory-defence(作業ディレクトリはリポジトリルート)

## 0. あなたの役割と作業単位

`godot/` の製品コードを、マルチプレイ本実装に耐える構造へ段階的に作り直してください。

- **この指示書は4つの作業パッケージ(WP1-1〜WP1-4)に分かれている。指示された1つのWPだけを実施すること。**
  (例: 「WP1-2を実施して」と言われたらWP1-2のみ。次のWPへ勝手に進まない)
- WPは順序依存があるため、必ず番号順に消化される。前のWPがマージ済みであることを`git log`で確認してから着手する。
- 1 WP = 1ブランチ = 1 PR。ブランチ名は `feature/wp1-1-foundation` のようにWP番号を含める。

## 1. 背景(必読ファイル)

1. `CLAUDE.md` — 開発ルール全般
2. `docs/14_godot-technical-spec.md` — §2(フォルダ)、§3(命名)、§4(Autoload)、§5(責務)、§6〜9(ネットワーク)、§10(Resource)、§13〜14(衝突レイヤー・InputMap)
3. `docs/13_godot-redevelopment-spec.md` — §11(Godot実装の前提構造)、§12.1(最小スライスのゴール)
4. `docs/16_combat-vehicle-balance.md` — 全バランス数値の正
5. `godot-spikes/spike_b_net/README.md` — 「本実装へ持ち込む判断」「持ち込まないもの」(WP1-4で必須)

## 2. 全WP共通の原則

- **作り直しは「in-placeの段階的リファクタリング」で行う。** 別プロジェクトを作らない。
  各WP完了時点で、ゲームは**ソロで最後まで遊べる状態を維持**する(現在の2Wave構成のまま)。
- 各WPの完了条件に必ず含まれる: スモークテスト合格
  (`..\.tools\godot-4.7\Godot_v4.7-stable_win64_console.exe --headless --path . -- --smoke-test` → `SMOKE_TEST_PASS`)
- 型付きGDScriptのみ。外部ライブラリ・アドオン禁止(docs/14 §19)
- バランス数値のロジック直書き禁止。既存の直書きも、触ったファイルでは残さない
- `unity/`・`prototype/`・`index.html`・`play/`・`godot-play/`・`godot-spikes/` は変更禁止
- アート素材(`godot/assets/art/`)は変更禁止(参照パスの変更は可)
- コミットメッセージは日本語。動かない状態でコミットしない
- 判断に迷ったらdocs/13・14を正とし、決められない場合は選択肢+推奨案を添えてPR本文で質問する

## WP1-1: フォルダ構成+Autoload骨組み+Resource基底+Catalog

### 内容

1. `godot/` のフォルダ構成を docs/14 §2 の形へ整備し、既存ファイルを移動する
   (`scripts/actors/` → 適切な配置へ。シーン・スクリプトの参照切れを全て直す)
2. Autoload 5種の骨組みを作る(docs/14 §4): `AppState` / `GameCatalog` / `NetworkSession` / `SaveService` / `AudioService`
   - この時点では最小実装でよい(NetworkSession/SaveService/AudioServiceは空に近くて可)
   - `AppState` にゲームバージョン定数(例: `GAME_VERSION := "0.1.0"`)を置く
   - **ラン固有状態をAutoloadに置かない**(docs/14 §4)
3. 全ゲーム定義Resourceの基底クラスを作る(docs/14 §10: id / display_name_key / description_key / icon / tags / content_version)
4. `GameCatalog` にResourceのID検索と検証を実装: ID重複・空ID・無効参照の全件検査。
   検証はスモークテストから呼び、違反があればFAILさせる
5. 衝突レイヤー名(docs/14 §13)とInputMapアクション名(docs/14 §14)をproject.godotに定義する
   (既存の独自アクション名があれば§14の名前へ移行)

### 受け入れ条件

- [ ] フォルダ構成がdocs/14 §2と一致し、参照切れゼロ
- [ ] Autoload 5種が登録され、起動エラーなし
- [ ] Resource基底+GameCatalog検証がスモークテストに組み込まれている
- [ ] InputMap・衝突レイヤーがdocs/14 §13〜14の名前で定義されている
- [ ] ソロで2Wave通しプレイ可能(挙動は変更前と同一)+スモークテスト合格

## WP1-2: Resourceデータ駆動化

### 内容

1. 現在ロジックに直書きされている数値・定義を、WP1-1の基底を継承したResourceへ移す:
   - 敵3種(ウォーカー/ランナー/クライマー)— 数値は docs/16 §4 の該当行を正とする
   - 武器(現行のプレイヤー射撃)— docs/16 §3 の基準武器
   - 車両(車体HP・外装HP・修理量)— docs/16 §8〜9
   - Wave定義(現行2Wave構成をデータ化。将来10Waveへ拡張できる形。構造は docs/16 §5 に合わせる)
   - レリック(現行3種をデータ化。効果はプロパティ駆動にし、種類追加がデータ追加で済む形。IDは docs/17 §2 の対応レリックに合わせる)
   - プレイヤー基本ステータス — docs/13 §3.6
2. `.tres` ファイルは `godot/data/<カテゴリ>/` に置く(docs/14 §2)。Resource IDはdocs/14 §3の規則
3. 現行実装の数値と docs/16 の数値が食い違う場合: **docs/16を正として採用**し、変更点をPR本文に列挙する
4. GameCatalogが全カテゴリを読み込み、検証対象に含める
5. **エクスポート安全なロードにすること(重要)。** WP1-1のGameCatalogは`DirAccess`で
   `res://data`を走査し拡張子`.tres`を拾う実装だが、Godotのエクスポート版では
   カスタムResourceが`.remap`等へ変換され、ランタイムのディレクトリ走査で`.tres`が
   見つからず**Catalogが空になる**既知の挙動がある。実データを入れる本WPで、
   次のどちらかで対処する: (a)`.remap`を考慮した走査にする、(b)明示マニフェスト
   (preload/`.tres`パス列挙リソース)方式へ切り替える。
   検証として、**Windowsエクスポート版を1つ書き出し、Catalogが正しく全件ロードされること**を
   確認しPR本文に記載する(D-11によりWeb版=ソロデモも同じロード経路を通るため)。

### 受け入れ条件

- [ ] 敵・武器・車両・Wave・レリック・プレイヤーの数値がすべて`.tres`にあり、ロジック直書きゼロ
- [ ] docs/16との数値対応表(またはdiff説明)がPR本文にある
- [ ] GameCatalog検証が全カテゴリをカバー
- [ ] **エクスポート版でCatalogが全件ロードされることを確認済み**(走査方式の場合は特に必須)
- [ ] ソロで2Wave通しプレイ可能+スモークテスト合格

## WP1-3: Run状態機械の分離

### 内容

1. `game_root.gd` の一極集中を解体し、docs/14 §5 の責務分割に沿って分離する。本WPの対象:
   - `RunCoordinator` — ラン状態機械(docs/13 §11.2)と各システムの調停
   - `StageDirector` — Wave時間・離脱(現行の準備/戦闘/報酬フェーズ進行)
   - `EnemyDirector` — 脅威予算とスポーン要求(現行のスポーン処理を移管。予算制は docs/16 §5 の構造で)
   - `VehicleState` — 車体・外装・共有資源(現行の車両HP/修理を移管)
   - `RewardSystem` — 戦果とレリック候補(現行の3択を移管)
2. ラン固有状態はRunシーンの子ノードが所有し、終了時に破棄(docs/14 §4)
3. 乱数を docs/14 §9 の形へ: run seedから `wave` / `reward` ストリームを派生。見た目用乱数と分離
4. 挙動はリファクタ前と同一を保つ(機能追加をしない)。ただしWave数・時間などが
   データ駆動(WP1-2)+状態機械で回ることを確認する

### 受け入れ条件

- [ ] game_root.gdが解体され、docs/14 §5の名前のクラスに責務が分かれている
- [ ] 1ファイル1責務・巨大クラスなし(目安: 1クラス300行以内)
- [ ] 同一シードで同じレリック候補が出る(乱数ストリーム分離の確認)
- [ ] ソロで2Wave通しプレイ可能(挙動は変更前と同一)+スモークテスト合格

## WP1-4: NetworkSession+複製層+ロビー最小

### 内容

**着手前に `godot-spikes/spike_b_net/README.md` の「本実装へ持ち込む判断」「持ち込まないもの」を精読すること。**

1. `NetworkSession`(Autoload)を実装: ホスト作成 / IP指定参加 / 切断 / peer管理(docs/14 §6)
   - **ソロプレイは`OfflineMultiplayerPeer`等でホスト単独として同一コードパスを通す**
     (ソロ=特別扱いの分岐だらけ、を作らない。またWeb書き出し(ソロ専用、D-11)が動く構成を保つ)
   - 接続時にバージョン照合(AppStateのGAME_VERSION不一致は拒否)
2. 複製層(`NetStateReplicator`相当)を実装。Spike Bの構造を製品分割で:
   - クライアントは「意図」のみ送信(入力sequence+移動軸+照準+射撃要求)。channel割当は docs/14 §7
   - ホストが移動・命中・敵・車両HP・修理を確定。スナップショット頻度は docs/14 §8
   - クライアント予測+ACK再適用補正、敵は100ms補間(Spike B判断1・3)
   - payloadはIDとプリミティブ値のみ(docs/14 §7)。Dictionary/可変長Arrayの安易な多用をしない(Spike B「持ち込まないもの」)
3. ロビー最小: タイトル→「ソロ開始 / ホスト作成 / 参加(IP入力)」→ 準備完了→ラン開始。
   見た目は仮でよい(グラフィックフェーズ対象外)
4. レリック選択の同期: 候補はホストが抽選し全端末へ配布。**本WPでは「ホストが選択確定」まででよい**(投票はPhase 2)
5. 動作確認: 同一PC2プロセスで、接続→2人移動→射撃→敵撃破→修理→2Wave通し→リザルトまで通す。
   確認手順(起動コマンド)をPR本文に記載する

### 受け入れ条件

- [ ] ソロ(オフラインピア)で従来どおり2Wave通しプレイ+スモークテスト合格
- [ ] 2プロセスで接続し、移動・射撃・敵・車両HP・修理・Wave進行・リザルトが同期する
- [ ] クライアントが結果値を送っていない(意図のみ。channel割当がdocs/14 §7どおり)
- [ ] バージョン不一致で接続拒否
- [ ] 2人での通しプレイ手順がPR本文にある
