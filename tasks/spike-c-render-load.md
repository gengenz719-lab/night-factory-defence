# 業務指示書: Spike C — 表示負荷の技術検証

発行: PM / 2026-07-11
対象リポジトリ: night-factory-defence(作業ディレクトリはリポジトリルート)
前提: Spike B(tasks/spike-b-enet.md)は合格・マージ済み。本タスクは独立しており、他の作業と並行できる。

## 0. あなたの役割

このプロジェクトの描画負荷検証を担当してください。
成果物は「動く検証コード」と「計測結果レポート」、そして**グラフィックフェーズの描画予算の提案**です。
製品コードの実装は今回のスコープ外です。

## 1. 背景(必読ファイル)

作業前に必ずこの順で読むこと:

1. `CLAUDE.md` — 開発ルール全般
2. `docs/14_godot-technical-spec.md` — 特に §1(採用技術。Compatibility renderer初期値)、§12(描画とパフォーマンス予算)、§17「Spike C: 表示負荷」
3. `docs/20_godot-roadmap.md` — 本タスクの位置づけ(Phase 3の検証ゲート。結果はグラフィックフェーズの予算になる)

このゲームは横視点2D。多層パララックス背景の上で敵・弾が大量に動く。
グラフィック本格投資の前に「どこまで盛れるか」の予算を数値で確定するのが本タスクの目的。

## 2. タスク内容

`docs/14_godot-technical-spec.md` §17 の Spike C を実施する:

- 背景5レイヤー(Parallax系。ダミー絵でよいが1920×1080を覆う実サイズのテクスチャを使う)
- 敵60体(移動AI+アニメ相当の変形/点滅程度でよい)
- 弾220(プレイヤー弾80+車載弾80+敵弾60。docs/14 §12の初期上限)
- GPUパーティクル(着弾・死亡演出相当を常時発生させる)
- 敵・弾はオブジェクトプールを使用する(docs/14 §12。プール有無の比較ができるとなお良い)
- 1920×1080でフレーム時間を記録し、60fps達成を判定する

### 計測条件

- レンダラー: Compatibility(初期値)。余力があればForward+との比較も歓迎(必須ではない)
- 記録する値: 平均fps、1% low(下位1%フレームの平均)、最大フレーム時間、
  オブジェクト数段階別(例: 敵15/30/60、弾55/110/220)の計測
- 計測はウィンドウ表示で行う(headlessはGPU描画されないため不可)。
  自動計測スクリプト(一定時間走らせてJSON出力)を用意し、手動起動→自動計測→自動終了にする
- 実行マシンのスペック(CPU/GPU/解像度)をレポートに記録する

## 3. 成果物

1. `godot-spikes/spike_c_render/` — 独立した最小Godotプロジェクト(既存の `godot/` とは分離)
2. `godot-spikes/spike_c_render/README.md` — 以下を含むレポート:
   - 起動・計測の再現手順
   - 計測結果表(段階別fps・1% low・最大フレーム時間、実行マシンスペック)
   - ボトルネック分析(何を増やすとfpsが落ちるか: 背景レイヤー/敵数/弾数/パーティクル)
   - **グラフィックフェーズへの描画予算提案**: 背景レイヤー数・同時パーティクル量・
     ライト数・1スプライトの推奨解像度の上限案(docs/14 §12の初期値を実測で裏付けor修正提案)
   - docs/14 §12への変更提案(あれば。提案のみ、docsは書き換えない)
3. `docs/20_godot-roadmap.md` のPhase 3にあるSpike C行のチェック更新(合格の場合のみ)

## 4. 受け入れ条件(Definition of Done)

- [ ] 背景5レイヤー+敵60+弾220+GPUパーティクルが1画面で動く
- [ ] 1920×1080・Compatibilityレンダラーで計測し、数値がJSONとREADMEに記録されている
- [ ] 60fps達成/未達の判定と、未達の場合は「何を削れば達するか」の分析がある
- [ ] 敵・弾がオブジェクトプールで管理されている
- [ ] グラフィックフェーズ向けの描画予算提案がREADMEにある
- [ ] `godot/` 以下の製品コードに一切変更がない

## 5. 制約・禁止事項

- **触ってよいのは `godot-spikes/spike_c_render/` と `docs/20` のチェック1行のみ。**
  `godot/`(製品コード)、`unity/`・`prototype/`(凍結参考実装)、
  `index.html`・`play/`・`prototype/`・`godot-play/`(公開配信物)は変更禁止
- 言語は型付きGDScriptのみ。外部ライブラリ・アドオンの追加禁止(docs/14 §19)
- Godot本体はリポジトリ内 `.tools/godot-4.7/` の実行ファイルを使用
  (`Godot_v4.7-stable_win64_console.exe`。無い場合は作業を止めて報告)
- スパイクコードは使い捨て前提。品質より検証速度を優先してよいが、レポートは丁寧に
- コミットメッセージは日本語・既存の履歴の書き方に合わせる

## 6. 進め方

1. ブランチ `spike/render-load` を作成して作業(mainへ直接pushしない)
2. 変更は小さくコミット(例: 「シーン雛形」→「敵・弾プール」→「背景・パーティクル」→「自動計測」→「レポート」)
3. 完了したらpushしてPRを作成し、最終レポート(下記フォーマット)を提出。マージはPM判断
