# 業務指示書: Spike B — 2人ENet通信の技術検証

発行: PM / 2026-07-11
対象リポジトリ: night-factory-defence(作業ディレクトリはリポジトリルート)

## 0. あなたの役割

このプロジェクトの通信技術検証を担当してください。
成果物は「動く検証コード」と「計測結果レポート」です。製品コードの実装は今回のスコープ外です。

1. `CLAUDE.md` — 開発ルール全般(AI向け共通ルール。あなたにも適用される)
2. `docs/14_godot-technical-spec.md` — 特に §1(採用技術)、§6〜9(状態所有権・RPC方針・更新頻度・乱数)、§17「Spike B: ネットワーク」

ゲームは「移動車両型・協力Waveサバイバル」(Godot 4.7 stable・型付きGDScript)。
1〜4人・プレイヤーホスト権威・ENetでのマルチプレイが仕様の心臓部であり、
本実装前に通信の成立性を使い捨てスパイクで検証するのが本タスクの目的です。

## 2. タスク内容

`docs/14_godot-technical-spec.md` §17 の Spike B を実施する:

- ホスト+クライアント1台の2プロセス構成(同一PC上の2インスタンス起動でよい)
- プレイヤー2体の移動(クライアント側は入力送信+予測、ホスト確定+補正)
- 射撃と命中判定(見た目は即時、ダメージ確定はホスト。docs/14 §6の「意図を送る」方式)
- 敵10体の同期(ホストがスポーン・AI・HPを所有、クライアントは表示)
- reliableとunreliableを別channelで使い分け(docs/14 §7のchannel割当に従う)
- 150ms RTT・パケット損失2%を模した条件で10分間、切断・状態不一致なく動作すること

### 遅延・損失の模擬方法

Godot標準APIにネットワーク劣化シミュレーションはない。実装方法は任せるが、
採用した方法(例: 送受信をラップする人工遅延/ドロップ層、外部ツール併用等)を
レポートに明記し、再現手順を残すこと。

## 4. 受け入れ条件(Definition of Done)

- [ ] 2インスタンスで接続し、移動・射撃・敵10体が動く
- [ ] 150ms RTT+2%損失の模擬条件下で10分間、切断・デシンクなし
- [ ] クライアントが結果値でなく「意図」を送る構造になっている(docs/14 §6)
- [ ] reliable/unreliableのchannel分離が実装されている(docs/14 §7)
- [ ] README.mdのレポートが再現手順込みで完成している
- [ ] `godot/` 以下の製品コードに一切変更がない

## 5. 制約・禁止事項

- **触ってよいのは `godot-spikes/spike_b_net/` と `docs/20` のチェック1行のみ。**
  `godot/`(製品コード)、`unity/`・`prototype/`(凍結参考実装)、
  `index.html`・`play/`・`godot-play/`(公開配信物。pushでPagesが自動デプロイされる)は変更禁止
- 言語は型付きGDScriptのみ。C#・外部ライブラリ・アドオンの追加禁止(docs/14 §19)
- Godot本体はリポジトリ内 `.tools/godot-4.7/` の実行ファイルを使用
  (`Godot_v4.7-stable_win64_console.exe`。Git管理外。無い場合は作業を止めて報告)
- スパイクコードは使い捨て前提。品質より検証速度を優先してよいが、
  レポートだけは本実装の判断材料になるため丁寧に書くこと(docs/14 §17「合格後」参照)
- コミットメッセージは日本語・既存の履歴の書き方に合わせる

## 6. 進め方

1. ブランチ `spike/enet-2player` を作成して作業(mainへ直接pushしない。ユーザーがアート作業でmainを使用中)
2. 変更は小さくコミット(例: 「接続確立」→「移動同期」→「射撃」→「敵同期」→「劣化条件」→「レポート」)
3. 動作確認は毎コミット前に実施。10分耐久は最後にまとめて実施でよい
4. 完了したらpushし、最終レポート(下記フォーマット)を提出。マージはPM判断なのでPRのままにする
