# AIバイブコーディングの回し方

AIを「3人目の開発メンバー」として扱う。役割は4つ。

| 役割 | 使い方 |
|---|---|
| 先生 | 分からない用語・エラーを聞く |
| 相棒 | 小さい機能を書いてもらう |
| レビュアー | 書いた(書かせた)コードを読ませて危ない所を指摘させる |
| 企画係 | レリック案・敵案・バランス調整案を量産させる |

## 大原則

```text
AIには「小さい作業」を渡す。
「このゲームを作って」ではなく「この1機能を、このファイルに、この条件で」。
```

このリポジトリには CLAUDE.md にAI向けルールが書いてあるので、
Claude Code / Cursor をリポジトリのフォルダで開けば、AIが自動的にプロジェクトの流儀を守る。

## プロンプト例(コピペして使う)

### 機能を作ってもらう

```text
godot/scripts/actors/enemy_unit.gd に「クライマーが屋根に張り付いている間、屋根外装に継続ダメージ」を足したい。
- ダメージ量などの数値はロジックに直書きせず、データ定義に置いて
- 型付きGDScriptで、既存のコードの書き方に合わせて、日本語コメント多めで
- 変更したファイルと、動作確認の手順を最後にまとめて
```

### エラーを直してもらう

```text
Godotでゲームを実行したらエラーが出た。

エラー文(Godotの出力/デバッガータブからコピペ):
(ここに貼る)

直前にやったこと:
(例: 敵のスポーン数値を変えた)

原因、直し方、再発防止を初心者にも分かるように教えて。
```

### バランス調整案を出させる

```text
docs/16_combat-vehicle-balance.md を読んで。
Wave 2 のクライマーが強すぎる気がする。敵予算と数値の調整案を3パターン、
それぞれの狙い(プレイヤーに何を要求する難しさか)つきで提案して。
```

### 週次クエストを作らせる

```text
docs/20_godot-roadmap.md を読んで。
初心者メンバーが2時間で達成できる今週の開発クエストを5つ、難易度★1〜5つきで出して。
```

### レビューさせる

```text
今回の変更差分をレビューして。
特に「マルチプレイ(ホスト権威、docs/14 §6〜8)で困る書き方」と
「データ定義に置くべき数値の直書き」がないかを見て。
```

## AIとの作業で守ること

1. AIが書いたコードは必ず一度読む。読めなければ「1行ずつ解説して」と頼む
2. 1回の依頼は1機能。大きな依頼は「まずタスクを分解して」から
3. 良かったプロンプトはDiscordの #aiに聞いたこと に貼って共有する
4. AIの提案がゲームの憲法([docs/13 §0](13_godot-redevelopment-spec.md))に反するときは断る
