# Spike B — 2人ENet通信の技術検証

Godot 4.7の`MultiplayerAPI`と`ENetMultiplayerPeer`で、プレイヤーホスト権威型の2人通信が成立するかを確認する使い捨てスパイクです。製品コード`godot/`からは完全に分離しています。

## 検証した構成

- ホスト1プロセス + クライアント1プロセス（同一Windows PC、localhost）
- プレイヤー2体。クライアントは移動入力を送ってローカル予測し、ホストの確定位置で補正
- 射撃はクライアントで即時描画し、照準の「意図」だけを送信。命中と敵HPはホストが確定
- 敵10体。スポーン、移動AI、HP、死亡、再スポーンはホストだけが更新
- 片道75ms（RTT 150ms）と2%メッセージ損失をアプリ層で模擬
- 物理60Hz、入力30Hz、プレイヤー状態20Hz、敵状態12Hz、敵補間100ms

## 起動方法

リポジトリルートで実行します。Godot本体は指定どおり`.tools/godot-4.7/Godot_v4.7-stable_win64_console.exe`を使います。

### 自動2プロセス検証（推奨）

次の1コマンドで2プロセスを起動し、600秒後に最終状態ハッシュを照合してJSONレポートを出します。PowerShellのシステム設定は変更しません。

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\godot-spikes\spike_b_net\run_two_instances.ps1 -DurationSeconds 600
```

出力先は`%TEMP%\spike-b-enet-<日時>\`です。`host.log`、`client.log`、`report.json`が作られます。短時間の再確認は`-DurationSeconds 10`、劣化なしとの比較は`-NoImpairment`を付けます。

### ホストとクライアントを個別に表示起動

2つのPowerShellで次を順に実行します。`--manual`を付けない場合は移動と射撃が自動入力されます。

```powershell
# ホスト
.\.tools\godot-4.7\Godot_v4.7-stable_win64_console.exe --path .\godot-spikes\spike_b_net -- --host --duration=600

# クライアント
.\.tools\godot-4.7\Godot_v4.7-stable_win64_console.exe --path .\godot-spikes\spike_b_net -- --client --duration=600
```

クライアントへ`--manual`を追加すると、WASD、マウス照準、左クリックで操作できます。ポートを変える場合は両方へ同じ`--port=29118`を指定します。別PCから接続する場合はクライアントへ`--address=<ホストIP>`も指定します。

## RPCと状態所有権

| channel | 転送モード | 内容 | 送信元 |
|---:|---|---|---|
| 0 | reliable | 初期状態、最終整合性検証 | ホスト（最終ACKのみクライアント） |
| 1 | unreliable ordered | 移動入力、照準、射撃要求 | クライアント |
| 2 | unreliable ordered | プレイヤー・敵スナップショット | ホスト |
| 3 | unreliable ordered | 射撃・命中の短命イベント | ホスト |

クライアントから送る値は入力sequence、移動軸、照準だけです。クライアント位置、命中した敵ID、敵HPなどの結果値は受け取りません。ホストは入力軸、入力順、射撃クールダウン、射線を検証して確定します。

クライアント移動は、未承認入力を履歴へ保存し、スナップショットのACK位置から未承認分を再適用します。敵は最後に届いたホスト位置へ100msを基準に補間します。

## 遅延・損失の模擬方法

`scripts/network_impairment.gd`がRPC送信をキューへ入れます。

- 全メッセージを送信側で75ms保留し、往復で約150msにする
- unreliableメッセージは送信前に2%を破棄する
- reliableメッセージは2%抽選時に破棄せず、ENet再送相当として追加1 RTT遅らせる
- ゲームルールとは独立した固定seedの乱数を使い、同じ役割では再現しやすくする

これはアプリケーションメッセージ単位の近似です。実パケットの分割、ジッター、並び替え、輻輳、ENetヘッダーを再現していません。したがって合格後も、別PC・実回線・外部ネットワーク劣化ツールで補完する必要があります。

## 10分耐久の計測結果

**結果: 合格**

2026-07-11、同一Windows PCのlocalhost上で自動2プロセス試験を実施しました。設定はRTT 150ms、メッセージ損失2%、600秒です。

| 項目 | 結果 |
|---|---:|
| 実走時間 | 600.017秒 |
| プロセス終了コード | ホスト0 / クライアント0 |
| 接続切断 | 0回 |
| 最終状態ハッシュ | 一致 |
| デシンク検出 | 0回 |
| プレイヤー / 敵 | 2体 / 10体 |
| ホスト確定射撃 / 命中 | 4,051回 / 1,282回 |
| 敵再スポーン | 250回 |
| クライアント平均補正 | 3.605px |
| クライアント最大補正 | 18.333px |
| ホスト側unreliable送信 / 模擬破棄 | 23,251件 / 451件（1.94%） |
| ホスト送信payload概算 | 4,381,128 bytes / 58.413kbps |

補正量は予測再適用後に測定しています。平均3.605px、最大18.333pxはプレイヤー半径18pxと同程度以下で、数値上は大きな巻き戻りにならない品質でした。ただし600秒試験はheadless自動入力であり、2人の主観的な操作感評価は未実施です。

payload値はホストの人工劣化キューへ積んだアプリデータの概算です。クライアント入力、UDP/ENetヘッダー、再送実バイトを含まないため、実回線帯域ではありません。ホスト側の観測破棄率1.94%は設定2%に対する固定seed抽選の結果です。

## 短時間の事前試験

600秒耐久前の8秒試験では、2プロセスとも終了コード0、最終状態ハッシュ一致、切断0、敵10体、命中16回でした。クライアントの移動補正は平均3.102px、最大9.060pxで、表示上の大きな巻き戻りはありませんでした。

## 本実装へ持ち込む判断

1. 入力sequenceとホストACKを必須にし、クライアント予測は未承認入力の再適用で補正する。
2. 初期状態・確定操作はchannel 0、入力は1、状態は2、短命イベントは3に固定し、RPC名だけでなくchannelもレビュー対象にする。
3. 敵AI・HP・命中はホストだけが更新し、クライアントは補間表示専用の状態を持つ。
4. 射撃要求が失われても次の入力を止めない。見た目の即時弾とホスト確定イベントを分離する。
5. テスト実行時は最終状態をreliableで収束させ、そのハッシュ、切断数、補正量、送信量を自動判定する。

## 本実装へ持ち込まないもの

- このスパイクの単一`main.gd`構造。製品版では接続、入力、状態複製、敵管理を分割する。
- アプリ層の人工劣化を実回線品質の完全な代替とみなすこと。
- `Dictionary`と可変長`Array`をそのまま大量に送る形式。製品版では用途別の小さいプリミティブpayloadを定義する。
- 最終同期で一致したことだけを、途中の表示品質まで保証する指標として扱うこと。
- クライアントが送った位置、命中、HPをホストが信用する書き方。

## docs/14 §6〜8への変更提案

- §7で「クライアントの射撃要求は入力意図としてchannel 1、ホストの射撃・命中通知はchannel 3」と送信方向を明記すると、channel 0 reliableへ誤って載せる実装を防ぎやすい。
- §8の通信スパイク指標へ、最終ハッシュ一致に加えて「予測再適用後の平均・最大補正px」と「アプリpayload概算kbps」を記録する、と追記する案を推奨する。

## 残課題

- localhost以外の実回線、ジッター、帯域制限、パケット並び替えの検証
- 3〜4人、途中参加、90秒再接続、ホスト切断の検証
- NAT越えとSteam Networking導入後の再計測
- ENet/UDPヘッダー込みの実帯域計測
- 製品版の敵数・弾数・Wave進行を含む負荷下での補正品質
